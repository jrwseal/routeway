# RouteWay Intelligence
### Logistics Intelligence Platform สำหรับ Route Optimization และ Carbon Footprint Tracking

---

## 1. ปัญหาที่แก้

ธุรกิจขนส่ง/กระจายสินค้าแบบ Milk Run (รถคันเดียววิ่งส่งหลายจุดแล้วกลับคลัง) ส่วนใหญ่ยังวางแผนเส้นทางด้วยประสบการณ์คนขับหรือ Excel — ทำให้:

- ใช้รถมากกว่าที่จำเป็น (แต่ละจุดส่งแยกเที่ยว หรือจัดกลุ่มไม่มีประสิทธิภาพ)
- ระยะทาง/เชื้อเพลิง/เวลาที่ใช้สูงกว่าที่ควรจะเป็น
- ไม่มีตัวเลขวัดผลด้าน CO₂ ที่อ้างอิงได้ ทำให้รายงาน ESG/ความยั่งยืนทำได้ยาก
- ไม่รู้ว่า "วิธีจัดเส้นทาง" แบบไหนดีที่สุดสำหรับข้อมูลจริงของตัวเอง ในแต่ละวันข้อมูลลูกค้าเปลี่ยน คำตอบที่ดีที่สุดก็เปลี่ยนตาม

## 2. โซลูชัน

RouteWay Intelligence เป็นเว็บแอปที่รับไฟล์ **CSV รายการลูกค้า/จุดส่ง** แล้ว:

1. คำนวณเส้นทางที่เหมาะสมด้วย **อัลกอริทึม VRP (Vehicle Routing Problem) 5 ตระกูล ทั้งหมด 8 variant**
2. รันทุก variant พร้อมกัน แล้วเปรียบเทียบผลลัพธ์แบบ side-by-side (ระยะทาง, ต้นทุน, CO₂, จำนวนรถ)
3. เลือกผลลัพธ์ที่ต้นทุนต่ำสุดให้อัตโนมัติ พร้อมให้ผู้ใช้สลับดูผลลัพธ์จาก variant อื่นได้ทุกตัว
4. แสดงผลผ่าน 5 มุมมอง: แผนที่เส้นทาง, พอร์ทัลคนขับแบบ step-by-step, กราฟลดคาร์บอน, สถิติรายคัน, ตารางเปรียบเทียบอัลกอริทึม

ไม่มีข้อมูล hardcode — ทุกอย่างคำนวณจากไฟล์ที่ผู้ใช้อัปโหลดจริง

## 3. Input / Output

**CSV ที่รับเข้า** ต้องมีคอลัมน์: `Location, Lat, Lon, Demand_Volume, Ready_Time, Due_Time` (รองรับคอลัมน์ `Weight` เสริม, รองรับไฟล์ภาษาไทย encoding Windows-874 อัตโนมัติ)

**Fleet Config** (ตั้งค่ากองรถ, ไม่บังคับ — มี Standard Pool default ให้ถ้าไม่ตั้ง): ต่อรถแต่ละคัน — Capacity (CBM), Fuel Consumption (L/km), Fixed Cost (บาท/เที่ยว), สี; และค่ากลาง — ค่าแรงคนขับ (บาท/ชั่วโมง), ราคาน้ำมันแยกตามคลาสรถ (4/6/10 ล้อ)

## 4. อัลกอริทึมทั้งหมด (จุดขายหลัก)

ระบบรันครบทั้ง 8 variant ทุกครั้งที่มีการคำนวณ แล้วให้ผู้ใช้เห็นว่าอัลกอริทึมไหนชนะในข้อมูลชุดนั้นจริง ๆ — ไม่ต้องเดา ไม่ต้องเลือกเอง

| # | อัลกอริทึม | ประเภท | หลักการ |
|---|---|---|---|
| 1-2 | **Clarke-Wright Savings** (มี/ไม่มี 2-opt) | Construction | รวมคู่ลูกค้าที่ "ประหยัดระยะทาง" มากที่สุดเข้าเป็นเส้นทางเดียวกันก่อน จนกว่าจะเต็มความจุรถหรือผิดช่วงเวลาส่ง — เป็น baseline มาตรฐานของอุตสาหกรรม VRP |
| 3-4 | **Nearest Neighbor** (มี/ไม่มี 2-opt) | Construction | จากจุดปัจจุบัน วิ่งไปหาลูกค้าที่ใกล้ที่สุดที่ยังไม่ได้ส่งเสมอ — เร็ว ตรงไปตรงมา เหมาะเทียบ baseline |
| 5-6 | **Sweep** | Construction | กวาดมุมรอบคลังสินค้า (คลัสเตอร์ลูกค้าตามทิศทาง) แล้วค่อยจัดลำดับในแต่ละคลัสเตอร์ — เหมาะกับพื้นที่ที่ลูกค้ากระจายเป็นวงรอบคลัง |
| 7 | **Or-opt + Simulated Annealing** | Local Search (built-in cleanup, ไม่มี 2-opt toggle แยก) | เริ่มจากผล Clarke-Wright แล้ว "ย้ายลูกค้าทีละกลุ่มเล็ก ๆ ข้ามเส้นทาง" 500 รอบ ยอมรับผลที่แย่ลงชั่วคราวแบบมีความน่าจะเป็น (จำลองการเย็นตัวของโลหะ) เพื่อหนีจาก local optimum — เป็นอัลกอริทึมเดียวที่ "ย้ายลูกค้าข้ามเส้นทาง" ได้ ตัวอื่นทำได้แค่จัดลำดับในเส้นทางเดียว |
| 8-9 | **Solomon I1 Insertion** (มี/ไม่มี 2-opt) | Construction, VRPTW-aware | ออกแบบมาสำหรับปัญหาที่มีหน้าต่างเวลาส่ง (Time Window) โดยเฉพาะ: เริ่มเส้นทางใหม่จากลูกค้าที่ไกลจากคลังที่สุด แล้วแทรกลูกค้าที่เหลือทีละคนโดยเลือกคนที่ "เสี่ยงตกขบวนมากที่สุด" (regret สูงสุด) ก่อน ป้องกันปัญหาลูกค้าไกล/เวลาบีบถูกทิ้งไว้ท้ายสุดจนวางเส้นทางไม่ได้ |

ทุก variant ผ่านการตรวจสอบความเป็นไปได้เดียวกัน (ความจุรถ + หน้าต่างเวลาส่ง Due_Time) ก่อนถูกนำมาเทียบกัน — Clarke-Wright Savings และ Or-opt + SA เป็น 2 อัลกอริทึมที่การันตีว่าเคารพ Due_Time ได้ 100%

**Post-process 2-opt**: ตัวเลือกเสริมสำหรับ 3 ตระกูลแรก+Solomon I1 — สลับคู่เส้นทางที่ตัดกันเพื่อลดระยะทางเพิ่มเติมหลัง construction

## 5. สูตรคำนวณ (โปร่งใส ตรวจสอบได้)

- **CO₂ (kg)** = `(2621 × Fuel Consumption[L/km] × Distance[km]) / 1000` — 2621 คือกรัม CO₂ เฉลี่ยต่อน้ำมันดีเซล 1 ลิตร (มาตรฐานงานวิจัยโลจิสติกส์)
- **ต้นทุนต่อเที่ยว** = ระยะทาง × อัตราสิ้นเปลืองเชื้อเพลิง × ราคาน้ำมัน + (เวลารอ/60) × ค่าแรงคนขับ + Fixed Cost ต่อคัน
- **การจัดรถต่อเส้นทาง**: Best-fit — เลือกรถที่เล็กที่สุดที่ยังจุความต้องการของเส้นทางนั้นได้พอดี (ลด CO₂/ต้นทุนโดยไม่ overpack)
- **จำนวนพาเลทมาตรฐาน** = ปริมาตรรวม (CBM) ÷ 1.2
- **% ประหยัด** = (ระยะทาง/CO₂ แบบเดิม − แบบ optimize) ÷ แบบเดิม × 100

## 6. หน้าจอหลัก (5 มุมมอง)

1. **Dashboard & Savings** — แผนที่เส้นทางแบบ interactive (Leaflet), KPI ระยะทาง/สถานะประหยัด, การใช้ความจุรถ, insight ภาษาคน ("ควรรวมออเดอร์เล็กเพิ่มถ้า utilization ต่ำกว่า 80%")
2. **Interactive Driver Portal** — จำลองมุมมองคนขับจริง: ปลายทางถัดไป, หน้าต่างเวลาส่ง, ปุ่มนำทาง Google Maps, step "กำลังไปส่ง → ส่งเสร็จแล้ว" ทีละจุดจนจบทัวร์
3. **Carbon Footprint** — กราฟเทียบ CO₂ ทุก algorithm variant พร้อม methodology ที่อ้างอิงได้
4. **Statistics Car** — เปรียบเทียบก่อน/หลัง optimize ทั้งระยะทางและคาร์บอน, breakdown ต่อคัน
5. **Algorithm Comparison** — ตารางเทียบทั้ง 9 แถว (8 อัลกอริทึม + Or-opt variant เดียว) real-time พร้อม popup แสดงอัตโนมัติหลังอัปโหลด และปุ่ม "View Algorithm" เรียกดูซ้ำได้จากแผนที่

## 7. Technical Stack

React 19 + TypeScript + Vite, Tailwind CSS 4, Leaflet/react-leaflet (แผนที่), Recharts (กราฟ), PapaParse (CSV), date-fns — Client-side ทั้งหมด ไม่ต้องมี backend/database สำหรับคำนวณเส้นทาง (ใช้ OSRM public API เสริมเมื่อมีอินเทอร์เน็ต, fallback เป็น Haversine formula ในตัวถ้าไม่มี)

## 8. จุดต่างจากคู่แข่ง/โซลูชันทั่วไป

- **เทียบหลายอัลกอริทึมพร้อมกันจริง ไม่ใช่ตัวเดียวตายตัว** — ข้อมูลแต่ละวันไม่เหมือนกัน อัลกอริทึมที่ดีที่สุดก็ไม่เหมือนกัน ระบบให้ผู้ใช้เห็นและเลือกได้ ไม่ต้อง "เชื่อ" ว่าอัลกอริทึมเดียวจะดีที่สุดทุกครั้ง
- **โปร่งใสเรื่องสูตร** — ทุกตัวเลข CO₂/ต้นทุนมีสูตรอ้างอิงเปิดเผยในแอป เหมาะกับงานที่ต้องรายงาน ESG หรือชี้แจงลูกค้า
- **ไม่ต้องมี IT infrastructure** — รันในเบราว์เซอร์ ไม่ต้องติดตั้ง server, ไม่ต้องส่งข้อมูลลูกค้าออกไปนอกเครื่อง (ยกเว้นเรียก OSRM เสริม)
- **ปรับ Fleet Config ได้ละเอียดถึงระดับรถแต่ละคัน** — ไม่ใช่แค่ตั้งค่าเดียวเหมารวมทั้งกองรถ
